## 2026-07-07 — [bash] EXIT trap can still crash on `local` vars under `set -u`

Why: fixing `merge_one()` in `.claude/scripts/cts-sync.sh` (temp-file cleanup on error exit), a `trap 'rm -f "$base" "$result"' EXIT` inside a function with `local base result` crashed with "unbound variable" the moment `set -e` aborted the function on a failed `cp`/`git show` — bash tears down the function's local scope _before_ running the EXIT trap, so the trap sees the names as genuinely undeclared, not just empty. Fix: make the trap's referenced variables function-scoped-but-not-`local` (plain global assignment) when the function isn't reentrant/recursive, so the trap always resolves them regardless of teardown timing. Caught only via a forced-failure test (chmod'd a real file read-only), not by reasoning about the code — a `trap ... RETURN` fix looked even more plausible on paper and was already reverted once for a different bug (shell-global re-firing on later unrelated returns). Belongs in (guess): rule (bash scripting gotchas, no dedicated rule file yet — maybe new `rules/shell-scripting.md` if this class of bug recurs)

## 2026-07-16 — [cts-sync] blank-payload-path collision causes silent overwrite of unrelated local files

Why: adding `.prettierignore` to `cts-payload.txt` (a brand-new payload entry, needed to stop a vendored skill from being reformatted — see the entry below) caused `cts-sync.sh update` to silently overwrite `penny`'s pre-existing, functionally-unrelated `.prettierignore` (its own build-output ignores: `/dist`, `/coverage`, `/.nx/cache`, `.angular`, `.nx/self-healing`) instead of flagging it as locally-modified. Root cause: `copy_one()`'s "is this locally modified" check diffs the target against the *old* `.cts-version` SHA's copy of that same payload path — for a path that has never been in the payload before, there is no old-SHA copy to diff against, so the check can't distinguish "file doesn't exist yet" from "file exists locally for unrelated reasons," and both fall through to unconditional overwrite. Caught by manual review, not the tool; reverted by hand (restored original entries, appended the new line). Belongs in (guess): rule/tooling — full fix scoped in `tasks/todo/2026-07-16-06-cts-sync-payload-overwrite-and-format-drift.md`, step 1.

## 2026-07-16 — [tooling] Prettier does not treat YAML frontmatter as fully opaque — it enforces blank-line spacing after multi-paragraph folded scalars

Why: mis-diagnosed a "double-blank-line bug" in 9 `SKILL.md` files this session (single-blank assumed correct, double-blank assumed to be drift) and "fixed" it with a regex collapse — this was backwards. Verified by resetting the files to git HEAD and running `prettier --write .` again: it *added* a blank line after `description: >-` (a folded YAML scalar spanning two paragraphs, English + Ukrainian) and before the next frontmatter key (`triggers:`) — proving Prettier's markdown parser inspects and reformats spacing inside frontmatter for this pattern, contrary to the common assumption that frontmatter is opaque/untouched. (A truly opaque case does exist separately: content inside fenced code blocks, e.g. YAML example snippets in a doc body, is never touched either way — confirmed by the same test producing no change there.) Belongs in (guess): no dedicated rule file for Prettier/markdown gotchas yet — discard once the false regex-fix pattern (`\n{3,}` → `\n\n`) is understood not to be a safe blind normalization for markdown files with frontmatter.

## 2026-07-16 — [cts-payload] `.claude/hooks/knowledge-capture-nudge.sh`'s `/tmp` marker prefix was hardcoded to a project name during `/cts-contribute` generalization

Why: `git log -p` on this file in claude-ts showed the `/tmp` marker-file prefix used for the once-per-session nudge cadence guard was literally `penny-kc-nudge-` before contribution, then mechanically renamed to `claude-ts-kc-nudge-` when generalized via `/cts-contribute` — a blanket find-replace of the project name, not a deliberate choice. Since the underlying uniqueness guarantee is Claude Code's own `session_id` (already globally unique across every repo on the machine, `/tmp` being shared), the project-name segment serves only human-readability when eyeballing `/tmp` — hardcoding *any* single project's name into shared payload content is wrong for every other consumer. Fixed to a generic `cts-kc-nudge-` prefix. Belongs in (guess): rule — a `/cts-contribute` reminder to check for project-name literals (not just obviously-project-specific paths/packages) when generalizing shell/config content, since a plain find-replace of "penny" → "claude-ts" produces a value that's still wrong, just wrong for a different, larger set of consumers.

## 2026-07-16 — [tooling] `prettier --check .` on claude-ts still found 8 tracked files unformatted after the proseWrap reflow

Why: task-08 consumer-validation prep ran a fresh `prettier --check .` (via a consumer's installed binary, since claude-ts itself carries no `package.json`/prettier install) ahead of `/cts-update`, expecting a no-op since `6ce4728` already reflowed the repo for `proseWrap: never`. Instead 8 tracked files (7 skill `SKILL.md`s + `CHANGELOG.md`) were still non-conforming — that reflow commit's exclusion list evidently missed files added/edited afterward, or the check/write scope wasn't the full tracked tree. Separately, `.claude/skills/postgres-best-practices/` is intentionally excluded from any prettier pass (vendored third-party skill with its own build/validate process, per its own `README.md`) but this exclusion is enforced only by manual judgment each time — there's no `.prettierignore` encoding it, so any future contributor running `prettier --write .` naively will reformat vendored content that shouldn't be touched. Belongs in (guess): rule/config — add a `.prettierignore` entry for `.claude/skills/postgres-best-practices/` so the exclusion survives without tribal knowledge; consider a periodic `prettier --check .` as part of the release/upstream-push gate to catch drift like this before it accumulates.
